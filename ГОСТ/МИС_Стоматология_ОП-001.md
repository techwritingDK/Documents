# МИС. Модуль «Стоматология». Описание программы

**Обозначение документа:** МИС.СТМ.ОП-001  
**Версия:** 2.1.0  
**Дата составления:** 02 апреля 2025 г.  
**Составил:** Отдел технической документации

## Аннотация

Настоящий документ является описанием программного модуля «Стоматология» медицинской информационной системы (далее — МИС), разработанного в соответствии с требованиями ГОСТ 19.402-78 «Единая система программной документации. Описание программы».

Документ предназначен для технических специалистов, системных администраторов, разработчиков и сотрудников, осуществляющих сопровождение программного обеспечения.

Модуль «Стоматология» обеспечивает автоматизацию клинических, административных и финансовых процессов стоматологических подразделений медицинских организаций.

---

## 1. Общие сведения

### 1.1. Обозначение и наименование программы

Полное наименование: «Модуль автоматизации стоматологической службы медицинской информационной системы» (сокращённое наименование — Модуль «Стоматология», МИС.СТМ).

Обозначение: МИС.СТМ v2.1.0.

Модуль является структурной составляющей медицинской информационной системы и функционирует исключительно в её составе.

### 1.2. Программное обеспечение, необходимое для функционирования

Для корректной работы модуля необходимо следующее программное обеспечение:

- **Операционная система:** Microsoft Windows Server 2019/2022 (64-бит) или Linux (Ubuntu 20.04 LTS / Debian 11 и выше) на серверной части; Microsoft Windows 10/11 (64-бит) — на рабочих станциях.
- **СУБД:** PostgreSQL версии 14.0 и выше.
- **Среда исполнения:** Java Runtime Environment (JRE) версии 17 LTS и выше (для серверной части); Google Chrome 110+, Mozilla Firefox 110+, Microsoft Edge 110+ (для веб-клиента).
- **Веб-сервер:** Nginx версии 1.22 и выше (для балансировки нагрузки и проксирования).
- **Брокер сообщений:** Apache Kafka 3.4 и выше (для интеграционного взаимодействия).
- **Основная платформа МИС** версии 5.0 и выше (обязательное условие).

### 1.3. Языки программирования

Серверная часть модуля реализована на языке программирования Java (версия 17 LTS) с применением фреймворка Spring Boot 3.x. Пользовательский интерфейс реализован на языке TypeScript с использованием фреймворка Angular 16. Скрипты автоматизации развёртывания написаны на Bash и Python 3.11.

## 2. Функциональное назначение

### 2.1. Назначение программы

Модуль «Стоматология» предназначен для автоматизации деятельности стоматологических подразделений медицинских организаций: стоматологических клиник, поликлиник, стационарных отделений, а также частных стоматологических кабинетов, работающих в составе многопрофильных медицинских центров.

### 2.2. Классы решаемых задач

Модуль обеспечивает решение следующих классов задач:

- **Управление расписанием и записью пациентов:** ведение журнала записи на приём, распределение нагрузки между врачами, управление временными слотами, информирование пациентов.
- **Ведение электронной медицинской документации:** формирование и хранение медицинских карт пациентов стоматологического профиля (форма № 043/у), ввод жалоб, анамнеза, объективных данных осмотра, описание стоматологического статуса.
- **Одонтограмма и зубная карта:** графическое отображение состояния зубного ряда с возможностью отметки патологий, проведённых манипуляций и плана лечения.
- **Планирование и учёт лечения:** составление планов лечения, фиксация выполненных манипуляций, учёт использованных материалов.
- **Финансовый учёт:** формирование счетов на оплату, учёт оказанных платных услуг, интеграция с кассовым модулем МИС.
- **Формирование отчётности:** создание статистических отчётов по видам лечения, посещаемости, выработке, использованию материалов.
- **Взаимодействие с лабораторной и рентгенологической службами:** направление на анализы и исследования, получение результатов.

### 2.3. Функциональные ограничения

При использовании модуля необходимо учитывать следующие ограничения:

- Модуль не является самостоятельным программным продуктом и не предназначен для эксплуатации вне инфраструктуры МИС.
- Функции телемедицины (дистанционные консультации) в данной версии не реализованы и выполняются отдельным модулем МИС.
- Формирование сведений для передачи в ТФОМС и ФСС осуществляется модулем «Страховая медицина»; Модуль «Стоматология» предоставляет для него первичные данные через внутренний API.
- Работа модуля без подключения к серверу баз данных не предусмотрена; автономный (офлайн) режим отсутствует.

## 3. Описание логической структуры

### 3.1. Алгоритм программы

Работа модуля построена по событийно-ориентированной модели. Основной цикл обработки включает:

1. Приём HTTP/HTTPS-запроса от клиентского приложения или внешней системы.
2. Аутентификацию и авторизацию запроса через модуль безопасности МИС (OAuth 2.0 / JWT).
3. Маршрутизацию запроса к соответствующему сервису бизнес-логики (Scheduling, EMR, Treatment, Billing, Report).
4. Выполнение бизнес-логики: валидация данных, работа с базой данных, формирование ответа.
5. Публикацию доменных событий в брокер сообщений Kafka (при операциях, затрагивающих другие модули МИС).
6. Возврат ответа клиенту в формате JSON (REST API) или в виде HTML-страницы (веб-интерфейс).

### 3.2. Используемые методы

В модуле применяются следующие методы и подходы:

- **Доменно-ориентированное проектирование (DDD)** — бизнес-логика сосредоточена в доменных сервисах и агрегатах.
- **Архитектурный паттерн CQRS** (Command Query Responsibility Segregation) — разделение операций чтения и записи для повышения производительности.
- **Event Sourcing** — сохранение истории изменений клинических данных в виде доменных событий.
- **RESTful API** — взаимодействие между клиентом и сервером по принципам REST (HTTP-методы GET, POST, PUT, DELETE, PATCH).
- **Оптимистичная блокировка (Optimistic Locking)** — предотвращение конфликтов при одновременном редактировании записей несколькими пользователями.

### 3.3. Структура программы и описание составных частей

| Компонент | Тип | Назначение |
|---|---|---|
| Scheduling Service | Микросервис | Управление расписанием, записью и напоминаниями пациентам |
| EMR Service | Микросервис | Ведение электронных медицинских карт, одонтограммы, анамнеза |
| Treatment Service | Микросервис | Планирование и учёт выполненных стоматологических манипуляций |
| Billing Service | Микросервис | Формирование счетов, учёт оплат, интеграция с кассой |
| Report Service | Микросервис | Генерация статистических и клинических отчётов |
| API Gateway | Шлюз | Единая точка входа, маршрутизация, балансировка нагрузки |
| Web Client (Angular) | Фронтенд | Пользовательский интерфейс (браузерный SPA) |
| Notification Worker | Фоновый сервис | Отправка SMS/email-уведомлений пациентам и персоналу |

### 3.4. Связи программы с другими программами

Модуль «Стоматология» взаимодействует со следующими компонентами МИС:

- **Модуль «Регистратура»** — получение и передача данных о пациентах, запись на приём.
- **Модуль «Лабораторная служба»** — направление на лабораторные исследования, получение результатов.
- **Модуль «Лучевая диагностика»** — направление на рентгенологические исследования, просмотр DICOM-изображений.
- **Модуль «Аптека / Склад»** — списание расходных материалов, запрос остатков.
- **Модуль «Страховая медицина»** — передача реестров выполненных услуг для формирования счетов в страховые организации.
- **Модуль «Кассовый учёт»** — фискализация платёжных операций, интеграция с онлайн-кассой (54-ФЗ).
- **ФРМО / ФРМР (внешние системы)** — передача сведений о пролеченных пациентах в рамках интеграции с ЕГИСЗ.

## 4. Используемые технические средства

### 4.1. Серверное оборудование

- **Сервер приложений:** процессор с частотой не менее 2,5 ГГц, 8 ядер и более; ОЗУ не менее 32 ГБ; дисковая подсистема на базе SSD объёмом не менее 500 ГБ.
- **Сервер баз данных:** процессор с частотой не менее 3,0 ГГц, 16 ядер и более; ОЗУ не менее 64 ГБ; RAID-массив SSD объёмом не менее 2 ТБ.
- **Сетевое оборудование:** коммутаторы с пропускной способностью не менее 1 Гбит/с; наличие резервных каналов связи.

### 4.2. Рабочие станции пользователей

Минимальные требования к рабочим станциям медицинского персонала:

- **Процессор:** Intel Core i3 (8-го поколения) / AMD Ryzen 3 или производительнее.
- **Оперативная память:** не менее 8 ГБ.
- **Монитор:** диагональ не менее 21,5 дюйма, разрешение не менее 1920×1080 пикселей.
- **Манипулятор:** мышь с двумя кнопками и колесом прокрутки.
- **Сетевой адаптер:** Ethernet 100/1000 Мбит/с.

### 4.3. Дополнительное оборудование

При необходимости к рабочей станции могут быть подключены:

- Интраоральная камера (с передачей изображений через TWAIN-интерфейс).
- Радиовизиограф / дентальный рентгеновский аппарат с поддержкой DICOM 3.0.
- Принтер для печати направлений, карт и счетов.

## 5. Вызов и загрузка

### 5.1. Способ вызова программы

Модуль «Стоматология» является серверным программным компонентом и запускается автоматически в составе МИС при старте платформенного окружения. Запуск серверных микросервисов осуществляется средствами оркестратора контейнеров (Docker / Kubernetes) командой:

```bash
kubectl apply -f mis-stomatology-deployment.yaml
```

Доступ к пользовательскому интерфейсу осуществляется через веб-браузер по адресу:

```
https://<hostname>/mis/stomatology
```

### 5.2. Входные точки в программу

Внешнее взаимодействие с модулем осуществляется через следующие входные точки:

- **REST API Gateway** — основная точка входа для веб-клиента и внешних интеграций: `https://<hostname>/api/v2/stomatology/`
- **Kafka Topics** — асинхронное получение событий от других модулей МИС: топики `mis.scheduling.events`, `mis.patient.updated`, `mis.lab.results`.
- **DICOM Service Endpoint** — приём диагностических изображений от радиовизиографов: порт 4242 (DICOM C-STORE).

### 5.3. Сведения об использовании памяти

Рекомендуемый объём ОЗУ для каждого экземпляра микросервиса — не менее 2 ГБ. Для кластерной конфигурации (3 реплики) минимально необходимый объём ОЗУ составляет 8 ГБ на каждом узле. Объём хранимых данных зависит от количества пациентов и срока эксплуатации системы.

## 6. Входные данные

### 6.1. Характер и организация входных данных

Входные данные модуля разделяются на три категории:

1. Данные, вводимые вручную медицинским персоналом через веб-интерфейс (интерактивные формы).
2. Данные, поступающие из других модулей МИС через REST API и Kafka (структурированные JSON-пакеты).
3. Диагностические изображения, поступающие с оборудования по протоколу DICOM.

### 6.2. Формат и описание входных данных

| Тип данных | Формат | Описание |
|---|---|---|
| Запись на приём | JSON | patientId, doctorId, slotStart, slotEnd, serviceCode, notes |
| Медицинская карта (043/у) | JSON | Структурированный объект с разделами: жалобы, анамнез, осмотр, диагноз (МКБ-10), план лечения |
| Одонтограмма | JSON / SVG | Массив зубов (ISO 3950), для каждого: состояние, диагноз, проведённые манипуляции |
| Манипуляции / услуги | JSON | serviceCode (прайс МИС), quantity, doctorId, tooth, materials[] |
| Данные о пациенте | JSON | ФИАС-адрес, СНИЛС, полис ОМС, телефон, e-mail |
| Рентген / DICOM | DICOM 3.0 | Серия изображений с метаданными (DICOM tags: PatientID, StudyDate, Modality) |
| Результаты анализов | HL7 FHIR R4 | Ресурс Observation: код исследования (LOINC), значение, единицы, референсный диапазон |

### 6.3. Предварительная подготовка входных данных

Перед вводом данных в модуль необходимо:

- Убедиться в наличии действующей регистрации пациента в МИС (уникальный идентификатор пациента — UUID).
- Обеспечить актуальный прайс-лист услуг, загруженный в модуль «Финансы» МИС.
- Настроить расписание врачей-стоматологов в модуле «Регистратура».
- Произвести сопоставление кодов услуг с классификатором МКБ-С (коды стоматологических услуг) в разделе «Справочники» МИС.

## 7. Выходные данные

### 7.1. Характер и организация выходных данных

Выходные данные модуля разделяются на следующие виды:

- **Экранные формы веб-интерфейса** — оперативное отображение информации пользователю.
- **Документы, формируемые для печати** — медицинские карты, направления, рецепты, счета.
- **Данные для других модулей МИС** — события и структуры данных, публикуемые через API и Kafka.
- **Аналитические отчёты** — файлы в форматах XLSX, PDF, CSV.

### 7.2. Формат и описание выходных данных

| Наименование | Формат | Описание |
|---|---|---|
| Медицинская карта стоматологического больного (ф. 043/у) | PDF / DOCX | Печатная форма истории болезни; подписывается врачом и пациентом |
| Одонтограмма | SVG / PDF | Графическое представление зубной карты с условными обозначениями |
| План лечения | PDF | Перечень рекомендуемых манипуляций с указанием стоимости и сроков |
| Счёт на оплату | PDF / JSON (фискальный) | Счёт с QR-кодом для оплаты; передаётся в онлайн-кассу (54-ФЗ) |
| Направление на исследование | PDF / HL7 FHIR R4 | Направление в лабораторию или рентген-кабинет |
| SMS / Email-уведомление | Текстовое сообщение | Напоминание о приёме; подтверждение записи |
| Статистический отчёт | XLSX / PDF / CSV | Аналитика по посещаемости, видам лечения, выработке врачей |
| API-ответ (REST) | JSON | Ответ на запрос внешней системы; структура соответствует OpenAPI 3.0-спецификации модуля |
| Kafka-событие | JSON (Avro) | Доменное событие для потребителей: AppointmentCreated, TreatmentCompleted, InvoicePaid и др. |

### 7.3. Способ кодирования выходных данных

Все текстовые выходные данные передаются в кодировке UTF-8. JSON-структуры соответствуют стандарту RFC 8259. Печатные документы формируются в кодировке Unicode (UTF-8). Двоичные файлы изображений передаются в соответствии со стандартом DICOM 3.0.
